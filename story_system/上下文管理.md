# 上下文管理机制

> 目的：防止写前必读的状态文件无限膨胀，撑爆上下文导致写作中途"失忆"、影响小说质量。
> 用法：**每次开新章前，先按本文件第一节做一次水位检查**，绿灯正常开写，黄/红灯按第三节处理。

---

## 一、每章开写前：水位检查（强制，我主动做）

只盯三个"只增不减"的累积大户，量它们的字符总和 = **水位线**。

检查命令（在 `story_system/` 目录下跑）：

```bash
cat 已发生事件.md 人物状态.md 章节进度表.md | wc -m | tr -d ' '
```

对照三个灯：

| 灯 | 三文件合计字符 | 动作 |
|----|--------------|------|
| 🟢 绿灯 | < 55,000 | 正常开写。我跟你说一句"上下文水位 OK"。 |
| 🟡 黄灯 | 55,000 – 80,000 | 预警。我提醒"水位偏高，建议本章写完后做一次瘦身"。本章仍可正常写。 |
| 🔴 红灯 | > 80,000 | 先瘦身再写。我会说"必须先归档，否则可能写到一半失忆"。 |

**基线记录（示例，请替换成你自己项目的实测值）：某次归档时三文件合计 = 36,108 字符（🟢 绿灯）。** 阈值 55,000 / 80,000 也只是经验值，字数增长速度因项目而异，可按自己项目实际情况调整。

---

## 二、为什么是这三个文件

- `已发生事件.md`、`人物状态.md`、`章节进度表.md` —— 每写一章都往上堆，是膨胀主因。
- 其余写前必读（写作风格指南、自检清单、流程、README、分卷大纲）体积**稳定不涨**，不用盯。
- 伏笔清单缓慢增长，但它是"活清单"（写作要随时查回收状态），不归档，只在它单独超过 12,000 字符时才考虑精简已回收的旧伏笔。

---

## 三、处理方法：归档，不删除

核心原则：**老章节的细节挪走，不丢；主文件只留一句话索引。**

### 操作步骤

1. **定归档范围**：取"已封存且最早的一批章节"（一次归档约 5–8 章，留最近 5 章细节在主文件里）。
   - 例：写到第 20 章触发瘦身 → 归档第 1–8 章，主文件保留第 9–20 章详细。

2. **`已发生事件.md`**：
   - 把要归档章节的逐条事件，整段剪切到 `已发生事件_归档.md`。
   - 主文件对应位置替换成一行索引：`### 第X章《标题》→ 一句话结果（详见 已发生事件_归档.md）`。

3. **`人物状态.md`**：
   - **"当前状态"区块永远留在主文件**（位置、情绪、伤势、秘密、物品——这些是活的，写作必用）。
   - 只把底部"逐章变化流水"里的老章节段落，剪切到 `人物状态_归档.md`，主文件留一句话索引。

4. **`章节进度表.md`**：
   - 一般不用归档（本身是索引性质、行很短）。只有它单独超过 15,000 字符时，才把很老章节的"详细备注"压成单行。

5. **绝不归档**：伏笔清单的未回收伏笔、人物状态的当前状态、写作风格指南、自检清单、流程、分卷大纲。这些是"活规则/活状态"，写作随时要用。

6. **归档后复测水位**，确认回到 🟢 绿灯，并更新本文件"基线记录"。

### 归档文件去哪读

归档文件**不进写前必读**。只有当某章要回收很老的伏笔、或要核对很老的情节时，才单独 Read 对应的 `_归档.md`。

---

## 四、和现有流程的关系

- 这是写前流程的"第 0 步"：**先水位检查 → 再读状态 → 查语料 → 给计划**。
- 瘦身动作属于"机械活"，用 low/medium effort 即可，不占创作档位。
- 归档完成后，照常更新 `章节总结/` 不受影响（总结本来就是分章独立文件，不膨胀）。
