# 章节自检清单（纯框架模板 · 写完每章必过 · 全过＝候选定稿）

> **维护原则·一条一义：** C 节每条只说一件事——一条塞两三个规则，AI 核对时会只答前半漏后半（实证教训）。另外**可机检的项别放这里**（进 style_gate 词表/规则），本清单只留脚本查不了的语义判断题。
> 用法：写完一章 → 逐项过 → 全过 = 候选定稿，AI 不再自己找茬回头雕（正式封存＝用户认可＋阶段5完成，见 写作流程.md 总纪律1）。
> 有不过的 → 只修那一项 → 重过。"满意"不是标准，"全过"才是。
> **2 轮封顶（停止条件）：** 同一项最多修 2 轮；2 轮仍不过 → 停手，列出卡点+试过哪两招+两个备选方向，交用户定，不自己硬改（防空转烧额度、越改越糟）。
>
> **说明：** 本清单只保留 A/B/C 三大块的结构和"检查哪些维度"的条目名。C 段每条的具体病例、具体改法、例句、频率数字均已清空，只留"检查方向"。请按自己项目的题材、机制、实测错题，把占位内容替换成自己的。

## A. 结构必查（consistency_review 六项）
- [ ] 剧情逻辑：关键事件有前提、关键决定有触发、关键变化有后果
- [ ] 人物目标：主角这章想干啥清楚，没无故漂移
- [ ] 情绪/关系：情绪有来路，关系温度接得上上一章
- [ ] 身体/信息：伤势、秘密、误会、已知信息没丢没穿帮
- [ ] 场景/转场：读者不迷路，上一场余力带进下一场
- [ ] 章末承接：收在"变化/钩子"上，下一章第一拍接得住

## B. 设定与伏笔回收（对照 story_system/）
- [ ] 出场人物言行符合人设（查 人物状态.md / 角色卡：性格、声音、关系）
- [ ] 该埋的伏笔埋了、该收的伏笔收了（查 伏笔清单.md）
- [ ] 用上了之前的钩子/线索（没有"建了设定却没用"的浪费）
- [ ] 没临时发明破坏既有设定的大转折

## C. 历史错题（别再犯 · 来自 troubleshooting，每错一行）

> **分类建议：** 按共性分组，逐项过时「按组扫」——先看每组的共性陷阱领读、再过组内条目。
> **分工自检 4-5 路 Agent 各领哪几类 → 见 `检查表/阶段4-写后自检.md` 底部映射。**
> **每章重点盯的高频重犯类：** （占位）此处填写你自己项目实测的高频错题类别排序，本系统不提供内容。

### 一 · 主角专属信息机制逻辑（如果你的项目有类似"信息差/心声/读心"的机制，这是穿帮重灾区）
- [ ] 检查"谁能感知到该信息"是否严格按设定执行，别让不该感知的角色感知到
- [ ] 检查该机制逻辑是否自洽：是否有人对主角"没说出口的信息"做出反应，而主角却毫无知觉
- [ ] 检查是否有敏感设定字眼在该机制里被直白暴露（如秘密身份被直接点破）

### 二 · 感情线角色与反差萌
- [ ] 检查正文是否点破了角色专属反应特征对应的具体设定/隐喻字眼（应彻底不点破，只靠动作神态暗示）
- [ ] 检查专属反应特征是否用反，或描写千篇一律（同一情绪反复用同一词组）
- [ ] 检查角色内心独白/语言风格是否符合人设，有没有走音变成不符合人设的腔调
- [ ] 检查配角是否被单薄证据轻易说服、轻信不利于主角的信息
- [ ] 检查角色呈现是否扁平、缺乏反差

### 三 · AI 味 · 写太满（AI 最爱"什么都说尽/说满"）
- [ ] 检查是否有蛇足点题句（呈现完又补一句总结/盖章句）
- [ ] 检查走心/沉重/虐/慌的段落是否收住了吐槽腔与调侃外号
- [ ] 检查比喻是否为工整/扣意象硬凑，机制术语是否摆得过于直白
- [ ] 检查含蓄处是否戛然而止，有没有多余的解释句/收尾动作冲淡留白
- [ ] 检查设定/伤情/机制等背景信息是否一次性铺陈过多
- [ ] 检查吐槽/口头禅/卖萌用语是否同一套路重复过频
- [ ] 检查是否存在文绉绉/装腔作势、不像大白话的句子
- [ ] 检查心理活动是否被叙述者代为解释，而不是让角色自己呈现

### 四 · 留白 · 钩子 · 反派（该藏的藏住，别替读者说尽）
- [ ] 检查前期白皮反派的阴面是否写得太满（是否用内心独白直接自白算计）
- [ ] 检查章末悬念钩子是否"露人"（把是谁/在想什么/什么意思替读者说了）
- [ ] 检查是否存在叙述者上帝视角点破/剧透，或反派/主角的计划被整套背出来
- [ ] 检查章末钩子是否只收在情绪上，缺外部拉力
- [ ] 检查章末是否留了悬空句，而不是明确的变化或钩子
- [ ] 检查核心宝物/大伏笔是否在前文被提前点名，该留到揭晓点才说破

### 五 · 核心设定一致性（如果你的项目有"先知/预知/重生"类设定，这是核心引擎自洽检查）
- [ ] 检查先知/预知类设定走出已知剧情范围后，是否还不合理地保持精准
- [ ] 检查"先知/记忆"类设定是否被误写成违反设定本身的物理动作（如把"记忆里的剧情"写成"翻阅实体书"）
- [ ] 检查该藏的核心机制是否被系统/旁白提前说破
- [ ] 检查角色战斗力/危险应对边界前后是否一致
- [ ] 检查武器/功法/装备体系是否前后统一
- [ ] 检查新设定/新角色/新势力/新道具登场前是否有铺垫，避免突兀出现

### 六 · 硬伤 · 常识（低级穿帮和硬伤，机械可判定，过稿前扫一遍）
- [ ] 检查是否有时代/背景错位（角色对不属于自己认知范围的事物反应不合理）
- [ ] 检查是否混入了不适合本项目基调的现实敏感元素
- [ ] 检查角色面对实体威胁时是否有即时的生理/动作反应，而不是直接跳到冷静分析
- [ ] 检查数字/数量描述是否与实际内容一致
- [ ] 检查多人在场对话是否点明了说话对象
- [ ] 检查标点（如破折号）使用是否克制，有没有滥用

### 七 · 读者视角（第 5 路 sub agent 专查 · 不靠清单靠直觉）
> **此节不逐条过。** 由独立 sonnet sub agent 以"第一次读者"身份通读本章正文，找以下四类问题：
- [ ] **一头雾水**：读到哪里觉得"角色凭什么知道这个？""前面没说过啊？"——角色知道了不该知道的信息
- [ ] **作者替读者下结论**：画面/动作已经说清楚了，后面又多一句总结——详见 `docs/troubleshooting/` 下相关笔记（若你的项目没有，可自建同名笔记）
- [ ] **读着别扭**：某句话要回翻/重读才看懂，指代不清，或句式拧巴
- [ ] **节奏卡壳**：某段太长读不下去 / 某处转折太突然没有过渡
